# MUSEY AI Lab 项目说明

> 版本 0.1 · 2026-03-17

---

## 一句话说清楚

MUSEY AI Lab 是一块"可以连线的画布"——把图片拖上去、写一段描述、选个模型，画布上就会长出新的图片或视频。节点之间可以互相引用，一张参考图能派生出一整条创作链。

## 1. 它能干什么

**画布** — 无限平移和缩放。双击空白处弹出菜单，可以新建图像节点或视频节点。多张图片可以一次拖进来，每张各自成为一个节点。

**图像生成** — 选中节点后底部弹出输入面板。写提示词，选模型、比例、分辨率、数量，点发送。生成的结果会出现在右侧新节点上，和源节点之间有一条虚线连着。提示词不会清空，可以直接改几个字再跑一次。

**参考图引用** — 节点之间拉一条线就能建立引用关系。输入框里按 `@` 会弹出已连接的参考图列表，选中后标签会高亮显示在提示词里。最多支持 6 张参考图同时喂给模型。

**视频生成** — 切换到视频模式后，调用 Veo 3.1 生成短视频。可以选分辨率（720p/1080p/4K）和时长（4s/6s/8s）。生成完成后直接在节点内播放，支持暂停和下载。

**语音输入** — 输入面板右下角有麦克风按钮。单击开始录音，再单击停止，语音会追加到已有的提示词后面。

**历史记录** — 左侧栏可以展开历史面板，按时间倒序显示每次生成的提示词、状态、耗时和错误信息。数据存在浏览器 localStorage 里，刷新不丢。

**中英切换** — 左侧栏底部用户菜单里可以切换界面语言。

## 3. 跑起来

**前置条件**
- Node.js >= 20
- 一个能用的 Google Gemini API Key

```bash
# 安装依赖
npm install

# 启动（默认 http://localhost:3000）
npm run dev
```

打开页面后，点左侧栏底部头像 → 在 API Key 输入框里填写你的 Key → 点保存图标。Key 只存在当前浏览器本地，不会上传。

## 4. 怎么用

### 新建节点
- 左侧栏点 `+` 按钮，选图像节点或视频节点
- 双击空白画布弹出快捷菜单
- 直接把图片文件拖进画布（支持多张）

### 生成图像
1. 选中一个节点（空的或有图的都行）
2. 底部弹出输入面板，写提示词
3. 选模型、比例、分辨率、数量
4. 按回车或点发送

空节点上生成：结果直接填进当前节点。如果数量 > 1，多出来的会在右侧新建节点。

有图的节点上生成：当前图片作为参考图，结果全部出现在右侧新节点上。

### 使用参考图
- 从一个有图的节点拉线到目标节点，建立引用
- 在输入框输入 `@`，从弹出列表中选择参考图
- 选中节点后左右两侧的 `+` 按钮也能快速新建关联节点

### 图片操作
- 单击：选中节点
- 选中后右下角：替换图片 / 查看大图 / 下载
- 选中后右上角：删除节点

## 5. 技术架构

```
浏览器（纯前端，无服务端）
  │
  ├── React 19 + Vite 6 + TypeScript
  ├── UI: Tailwind CSS v4 + Lucide React + Motion
  ├── 画布: @xyflow/react (React Flow)
  │
  └── AI 调用
       ├── 图像: Google GenAI SDK → Gemini 系列模型
       └── 视频: Google GenAI SDK → Veo 3.1 (异步轮询)
```

几个值得说的设计选择：

**纯前端架构** — 没有后端。API Key 存 localStorage，模型调用直接从浏览器发出。好处是部署简单（静态站点就行），坏处是 Key 暴露在前端，只适合内部使用或个人使用。

**React Flow 做画布** — 节点位置、连线关系、拖放交互全交给 @xyflow/react。自定义了 ImageNode 和 VideoNode 两种节点类型。节点间的连线验证限制了"只能图像节点连图像节点"。

**并行生成** — 批量生成时（数量 > 1），对每个目标节点发起独立的 API 请求，用 `Promise.all` 并行等待。每个请求有独立的历史记录条目。

**视频轮询** — Veo 的视频生成是异步的。发起请求后每 10 秒轮询一次操作状态，最多等 10 分钟。完成后下载视频 blob 并生成 Object URL 显示在节点里。

**状态管理** — 没用 Redux 或 Zustand。节点数据用 React Flow 自带的 `setNodes` / `updateNodeData` 管理，历史记录用 Context + localStorage 持久化。当前规模下够用。

## 6. 代码结构

```
src/
  main.tsx                          应用入口，挂载到 DOM
  App.tsx                           顶层布局：Header + Sidebar + Canvas

  components/
    FlowCanvas.tsx                  画布核心
                                    - 节点/边的状态管理
                                    - 图像和视频生成逻辑（handleGenerate）
                                    - 拖放导入、双击新建、连线验证
                                    - 参考图收集（getReferenceImages）

    Sidebar.tsx                     左侧工具栏
                                    - 新增节点按钮
                                    - 资源库占位（商品/模特/背景/姿势/模板）
                                    - 历史记录面板
                                    - 用户菜单（API Key、语言切换、版本号）

    HistoryContext.tsx               生成历史的 Context Provider
                                    - addRecord / updateRecord
                                    - localStorage 持久化

    nodes/
      ImageNode.tsx                 图像节点（约 990 行，是最复杂的组件）
                                    - 空状态上传 / 有图状态展示
                                    - 浮动输入面板（提示词、模型选择、参数）
                                    - @ 引用系统（mention state machine）
                                    - 语音输入（Web Speech API）
                                    - 大图预览（Portal）
                                    - 左右 + 按钮（创建关联节点）

      VideoNode.tsx                 视频节点
                                    - 视频播放控制（play/pause/progress）
                                    - 上传 / 删除 / 下载
                                    - 加载和错误状态

  services/
    ai.ts                           图像编辑 API 封装（generateImageEdit）

  utils/
    geminiApiKey.ts                 localStorage 读写 API Key
    buildInfo.ts                    版本号和构建号
```

## 7. 接入的模型

| 前台名称 | 模型 ID | 用途 |
|----------|---------|------|
| Nano Banana 2 | `gemini-3.1-flash-image-preview` | 图像生成（默认） |
| Nano Banana | `gemini-2.5-flash-image` | 图像生成 |
| Nano Banana Pro | `gemini-3-pro-image-preview` | 图像生成（高质量） |
| Veo 3.1 Preview | `veo-3.1-generate-preview` | 视频生成 |

图像生成走 `generateContent` 接口，把参考图和提示词一起扔进 `contents`。视频生成走 `generateVideos` 接口，异步轮询直到 `operation.done`。

## 8. 数据和安全

- API Key 存在 `localStorage`，key 名是 `gemini_api_key`。不会发到任何服务端。
- 生成历史存在 `localStorage`，key 名是 `muse-history`。只存提示词和状态，不存图片数据。
- 图片以 base64 Data URL 形式保存在 React 状态中，刷新页面就丢。视频以 Blob URL 形式存在，同样刷新即失。
- 当前没有用户认证系统。

**如果要对外开放使用**，需要把模型调用迁移到服务端，不能让用户的浏览器直接拿着 Key 调 Gemini API。

## 9. 已知限制

- 画布状态不持久化。刷新页面后所有节点和图片都没了。
- 没有撤销/重做。
- 侧栏的商品库、模特库、背景库、姿势库、模板库目前是占位按钮，点了没反应。
- 视频节点的上传功能限制 5MB，大视频传不了。
- 语音输入依赖浏览器 Web Speech API，部分浏览器不支持。
- 生成出错后只能在节点上点"重试"，没有全局的错误恢复机制。

## 10. 迭代协作框架

这部分给内部团队用。定义产品和技术侧怎么协作推进。

### 10.1 阶段划分

| 阶段 | 目标 | 交付物 |
|------|------|--------|
| **P0 — 当前** | 可交互原型，验证节点式 AI 创作流程 | 画布 Demo（已完成） |
| **P1 — 可用** | 画布状态持久化、撤销重做、错误处理完善 | 可日常使用的工具 |
| **P2 — 可开放** | 服务端代理 API 调用、用户认证、资源库接入 | 多人可用的在线工具 |
| **P3 — 可扩展** | 插件系统、自定义节点类型、批量任务队列 | 平台级产品 |

### 10.2 需求流转

```
产品侧提 Issue（带场景描述和验收标准）
  → 设计侧出交互稿（如果涉及新交互）
  → 技术侧评估复杂度，拆成可执行的子任务
  → 开发 → PR Review → 合并
  → 产品侧验收
```

每个 Issue 打标签标明阶段（P0/P1/P2/P3）和类型（feature / bug / refactor / infra）。

### 10.3 技术侧迭代原则

- **先跑通再优雅。** 当前阶段优先验证体验，代码结构可以在 P1 做重构。
- **FlowCanvas.tsx 需要拆分。** 目前画布核心和生成逻辑混在一个 760 行的文件里。P1 阶段应该把生成逻辑抽成独立的 hook 或 service。
- **ImageNode.tsx 是最重的组件。** 将近 990 行，包含了输入面板、@ 引用、语音输入、大图预览等多个子系统。P1 阶段拆成子组件。
- **状态管理迟早要升级。** 目前用 React Flow 自带的状态 + Context 勉强够用。如果要加持久化、协作编辑，需要引入更正式的状态层。
- **测试为零。** 当前没有任何自动化测试。P1 阶段至少补上关键路径的集成测试。

### 10.4 设计侧迭代原则

- 新功能先出线框，和开发确认可行性后再出高保真。
- 交互规范写在 Issue 里，不要只截图——写清楚状态变化和边界情况。
- 当前深色主题是唯一主题，短期内不考虑浅色模式。

## 11. 部署

纯静态站点，可以直接部署到 Vercel / Netlify / Cloudflare Pages。

```bash
npm run build   # 产物在 dist/
```

Vite 构建配置做了手动分包：`@google/genai`、`@xyflow/react`、`lucide-react + motion` 各自独立 chunk，方便缓存。

版本号从 `package.json` 读取，构建号从 `GITHUB_RUN_NUMBER` 环境变量读取。页面左下角用户菜单里能看到。

再说一遍：不要把 Gemini Key 写进构建环境变量。保持用户自己在浏览器里填 Key 的模式。
